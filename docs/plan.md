# PDF Splitter CLI 改善計画

## 1. 目的
本計画は、このプロジェクトを「試作スクリプト」から「日常利用できる汎用CLIツール」へ引き上げるための実行計画を示す。

- **対象**: PDFアウトラインを持つ技術書全般（特に O'Reilly 形式）
- **成果**: 再現性・安全性・使いやすさ・保守性を備えた CLI
- **価値**: 個人利用だけでなく、チーム内運用や自動化パイプラインにも適用可能

---

## 2. 一般的なCLIツールのあり方（設計原則）

有用なCLIには、機能の多さ以上に「予測可能性」と「自動化親和性」が求められる。本ツールの設計にも次を適用する。

### 2.1 UX原則
1. **明確な入出力契約**
   - 引数・出力先・戻り値（exit code）が一貫していること。
   - エラー時に何が起きたか、次に何をすべきかを具体的に表示すること。
2. **安全なデフォルト**
   - 上書き抑止、確認付き削除、`--dry-run` による事前確認。
3. **段階的な高度化**
   - 初心者は `pdf-split input.pdf` のみで実行可能。
   - 上級者は `--max-depth`、`--include`、`--exclude`、`--config` などを活用可能。
4. **機械可読性**
   - 人間向けログと機械向け出力（JSON）を切り替え可能にし、CIや他ツール連携を容易にする。

### 2.2 運用原則
1. **冪等性と再実行性**
   - 同一入力・同一オプションなら同一結果を得られる。
2. **可観測性**
   - 進捗表示、処理件数、スキップ理由、失敗理由を記録。
3. **診断容易性**
   - `--verbose` / `--quiet` / `--log-file` / `--debug` を整備。
4. **後方互換性**
   - オプション仕様変更は段階的に行い、非推奨警告を経て破壊的変更する。

---

## 3. 現状評価（要約）

すでに以下は強みとして成立している。

- `argparse` によるCLI基盤
- 再帰的アウトライン分割
- 同一ページ競合の解決
- ファイル名サニタイズ
- 単体テストの基盤

一方で、実運用CLIとしては以下が不足している。

- 選別機能（Front Matter除外など）の柔軟性
- 実行計画の事前確認（dry-run）
- 設定ファイルによる再利用性
- ログ体系・終了コード体系の明文化
- 配布・CI・リリース手順

---

## 4. 実装ロードマップ（優先順）

## フェーズA: CLIとしての信頼性向上（最優先）

### A-1. Dry Run
- `--dry-run` を追加し、実際に分割せず以下を表示。
  - 分割単位（タイトル、開始/終了ページ）
  - 出力ファイル名
  - スキップ予定項目と理由
- 目的: 破壊的処理前の確認と利用者の安心感。

### A-2. Exit Code設計
- 例:
  - `0`: 正常終了
  - `2`: 引数エラー
  - `3`: 入力PDF不正/読込不可
  - `4`: 出力書込エラー
  - `5`: アウトライン不在または分割不可
- 目的: シェルスクリプト/CIでの制御を容易にする。

### A-3. ログと表示レベル
- `--verbose`, `--quiet`, `--log-file` を整備。
- 標準出力は要点のみ、詳細はログへ。

---

## フェーズB: 分割戦略の高度化（本プロジェクトの核）

### B-1. include/excludeフィルタ
- `--include-regex` / `--exclude-regex`
- `--ignore-case`
- 優先順位（exclude優先など）を明文化。

### B-2. Front/Back Matterポリシー
- `--front-matter {keep,merge,skip}`
- `--back-matter {keep,merge,skip}`
- 既定判定語（preface, foreword, index, colophon など）を定義し、上書き可能にする。

### B-3. 階層処理ポリシー
- `--split-level {part,chapter,section,auto}`
- `--flatten`
- `--max-depth` と整合するルールを文書化。

---

## フェーズC: 設定・再利用性

### C-1. 設定ファイル
- `--config path.{yaml,json}`
- ルールセットを保存し、書籍ごとに再利用可能にする。
- 優先順位: CLI引数 > 設定ファイル > デフォルト。

### C-2. サンプル設定集
- `configs/oreilly.yaml` などを提供。
- READMEに「典型ユースケース」を掲載。

---

## フェーズD: 配布品質

### D-1. パッケージ化
- `pyproject.toml` を導入し、`pipx` でインストール可能にする。
- エントリポイント（例: `pdf-split`）を整備。

### D-2. CI/CD
- GitHub Actions:
  - `pytest` 実行
  - lint（ruff or flake8）
  - type check（mypy）
- タグ運用によるリリース手順を定義。

### D-3. ドキュメント
- READMEに以下を明記:
  - インストール方法
  - 最小コマンド
  - よくある失敗と対処
  - 戻り値一覧
  - 設定ファイル例

---

## 5. テスト戦略

1. **ユニットテスト**
   - サニタイズ、アウトライン処理、フィルタ判定、ページ範囲算出。
2. **統合テスト**
   - 小規模サンプルPDFで実際に分割し、件数・ファイル名・ページ数を検証。
3. **CLI E2Eテスト**
   - サブプロセス経由でexit code、stdout/stderr、生成物を確認。
4. **回帰テスト**
   - 既知の難ケース（同一ページ競合、欠損アウトライン、重複タイトル）を固定化。

---

## 6. 非機能要件

- **性能**: 大型PDFでもメモリ過剰消費を避ける。
- **移植性**: Linux/macOS/Windowsで同一挙動。
- **信頼性**: 途中失敗時の中間成果物を制御（ロールバックまたは継続ポリシー）。
- **セキュリティ**: パス処理の安全性（パストラバーサル抑止、危険文字の除去）。

---

## 7. マイルストーン案

- **M1（短期）**: `--dry-run`, exit code, verbose/quiet, log-file
- **M2（中期）**: regexフィルタ、front/back matterポリシー、split-level戦略
- **M3（中期）**: config対応（YAML/JSON）+ サンプル設定
- **M4（中長期）**: パッケージ配布、CI/CD、ドキュメント完成

---

## 8. 完了判定（Definition of Done）

以下を満たした時点で「有用なCLIツール」と判定する。

1. 初見ユーザーがREADMEのみで基本分割を完遂できる。
2. 再実行時に同一結果が得られる（冪等性）。
3. エラー原因が明確で、exit codeで自動化可能。
4. フィルタ・階層・設定ファイルで主要な実運用要件をカバー。
5. テストとCIにより品質を継続担保できる。
